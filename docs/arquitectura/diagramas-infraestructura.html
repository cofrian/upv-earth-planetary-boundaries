UPV-EARTH — Mockup: Infraestructura y procesos

Diagramas para defensa técnica del TFG. Todo corre nativo en una sola máquina (sin Docker).

Leyenda de colores:   🟦 Frontend (navegador / Next.js)  ·  🟩 Backend (FastAPI + Python)  ·  🟪 LLM (Ollama + Qwen 2.5 14B)  ·  🟧 Almacenamiento (SQLite + ficheros)  ·  🟨 Modelo de embeddings (SPECTER en RAM)

1 · Infraestructura: 3 procesos en una sola máquina

Lo que NO está corriendo (aunque exista en el repo)

❌ Docker / Docker Compose
Existe docker-compose.yml con 5 servicios, pero la máquina no soporta Docker.
❌ PostgreSQL
psycopg2-binary está en requirements pero SQLAlchemy abre SQLite local.
❌ Nginx
El frontend Next.js hace de proxy al backend mediante rewrites internos.
❌ vLLM / servidor de chat separado
No hay nada en :8001. El chat reusa el mismo Ollama via API OpenAI-compatible.

2 · Proceso: subir un PDF y clasificarlo en un Planetary Boundary

Cada Stage emite un IngestionEvent (auditoría)

Cada paso del pipeline guarda en la tabla ingestion_events un registro con: event_type, event_payload (modelo usado, chars procesados, duración, etc.) y timestamp. Esto permite reconstruir QUÉ pasó en CADA paper aunque pase tiempo, y debuggear errores específicos sin tener que reprocesar.

3 · Proceso: chat RAG · responder pregunta sobre un paper

Por qué este chatbot es RAG y no agentic / fine-tuning

  • El LLM NO decide qué buscar — el backend recupera el contexto antes de hablarle al modelo.
  • El LLM NO genera PBs ni similares — eso ya está calculado en BBDD y se le pasa como contexto.
  • Trazabilidad: cada respuesta del LLM está "anclada" al texto que se le inyectó. Sin alucinaciones porque solo dispone de ese contexto.
  • Actualizable: añadir papers nuevos al índice tarda segundos, no requiere re-entrenar nada.
  • Mismo modelo que el pipeline (qwen2.5:14b en Ollama) → un solo modelo residente en VRAM, dos modos de uso.
